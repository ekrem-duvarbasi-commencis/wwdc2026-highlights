Siri becomes personal, contextual, and app-aware.

Siri now works with Apple Intelligence, App Intents, and app-provided schemas so it can understand user context and act across apps more naturally.

Personal Intelligence across apps

By adopting App Intents schemas, your app's content enters the Spotlight semantic index, enabling personal context understanding with attribution back to your app — and its actions become available through natural language without fixed trigger phrases.

  • Entity schemas contribute your app's content to the Spotlight semantic index so Siri can understand and surface it with attribution back to your app.
  • Intent schemas let people act on that content naturally — no specific phrases to define, and no code changes needed as Siri's language understanding evolves.
  • The View Annotations API adds on-screen awareness: map your views to entities so people can reference and act on whatever is visible on screen conversationally.

App Schemas make app content actionable

Apps describe entities and actions through schemas so Siri, Shortcuts, and Spotlight can reason about real app content instead of fixed trigger phrases. No specific phrases to define — the system handles language understanding and it improves over time without any code changes.

  • Important app objects need stable identities and readable names so Siri and Spotlight can find the same things users see in the interface.
  • Smaller, composable intents work better because the system can combine them with other app and system actions.
  • Set unsupported schema properties to nil — the system gracefully ignores them and still surfaces your content.
▶ Show code example
1 · Entity Schema

Conform your model to @AppEntity(schema:) so Siri and Spotlight understand what your content represents. Set unsupported properties to nil — the system gracefully ignores them.

@AppEntity(schema: .photos.asset)
struct PlantPhotoEntity: IndexedEntity {

    var displayRepresentation: DisplayRepresentation {
        DisplayRepresentation(
            title: "\(photo.name)",
            subtitle: "\(photo.creationDate.formatted(date: .abbreviated, time: .omitted))"
        )
    }

    static let defaultQuery = PlantPhotoEntityQuery()
    var id: Int { photo.id }

    // Required schema properties
    var creationDate: Date? { photo.creationDate }
    var assetType: PhotoAssetType? { .photo }
    var isFavorite: Bool { photo.isFavorite }
    var isHidden: Bool { photo.isHidden }

    // Not supported by this app — set to nil
    var location: PlaceDescriptor? { nil }
    var aperture: Double? { nil }
    var filter: PhotoFilterEffectType? { nil }

    var photo: PlantPhoto
    init(photo: PlantPhoto) { self.photo = photo }
}
2 · Intent Schemas

Conform your intent to @AppIntent(schema:) so people can trigger it through natural language — no fixed phrases needed. Use isAssistantOnly = true when transitioning away from an older intent.

@AppIntent(schema: .photos.openAsset)
struct OpenPlantPhotoIntent: OpenIntent {
    var target: PlantPhotoEntity

    func perform() async -> some IntentResult {
        // Navigate to the photo in the app
        return .result()
    }
}

@AppIntent(schema: .photos.createAssets)
struct CreateAssetsIntent: AppIntent {
    // Hidden from Shortcuts during transition from an older intent
    static let isAssistantOnly: Bool = true

    @MainActor
    func perform() async throws -> some ReturnsValue<[PlantPhotoEntity]> {
        // Create and return new photo entities
    }
}
3 · Spotlight Indexing

Donate entities to a named CSSearchableIndex so the system can surface them. Implement IndexedEntityQuery to handle reindexing requests from the system.

static func donatePhotos(modelData: ModelData) async throws {
    let entities = await modelData.plantPhotoEntities
    try await CSSearchableIndex(name: "PlantJournal_Photos")
        .indexAppEntities(entities)
}

struct PlantPhotoQuery: IndexedEntityQuery {
    func reindexEntities(
        for identifiers: [PlantPhotoEntity.ID],
        indexDescription: CSSearchableIndexDescription
    ) async throws {
        let photos = try await photoStore.fetch(ids: identifiers)
        try await CSSearchableIndex(name: "PlantJournal_Photos")
            .indexAppEntities(photos)
    }

    func reindexAllEntities(
        indexDescription: CSSearchableIndexDescription
    ) async throws {
        let all = try await photoStore.fetchAll()
        try await CSSearchableIndex(name: "PlantJournal_Photos")
            .indexAppEntities(all)
    }
}

AppIntentsTesting validates Siri, Spotlight, and Shortcuts out-of-process

AppIntentsTesting is a new framework (iOS 27+, currently in beta) that lets you execute intents, query entities, and verify Spotlight indexing and view annotations through the exact same infrastructure that Siri and Shortcuts use — without UI automation.

  • Tests run in a UI Testing target against a live app process. IntentDefinitions(bundleIdentifier:) is the single entry point; every intent, entity, and enum is accessed by string name, so the test target never needs to import app types.
  • Covered scenarios include intent execution, entity queries, chaining results between multiple intents, Transferable conformance, Spotlight indexing (spotlightQuery(_:)), and on-screen view annotations (viewAnnotations()).
  • Test-only intents marked isDiscoverable = false and wrapped in #if DEBUG replace fragile UI navigation for test setup and teardown.
▶ Show code example
1 · Test target setup

Launch the app and create an IntentDefinitions instance with your bundle ID. This discovers every intent, entity, enum, and query your app registers — no imports of app types needed.

class AppIntentsTests: XCTestCase {
    let app = XCUIApplication()
    var definitions: IntentDefinitions!

    override func setUp() async throws {
        app.launch()
        definitions = IntentDefinitions(
            bundleIdentifier: "com.example.my-app"
        )
    }
}
2 · Run an intent & query entities

Look up an intent by name and call run(). The result supports @dynamicMemberLookup, so you read return-value properties by name. Entity queries work the same way — entities(matching:) simulates how Siri resolves spoken names.

// Run an intent
func testOpenFavorites() async throws {
    let result = try await definitions
        .intents["OpenFavorites"]
        .makeIntent()
        .run()
    XCTAssertEqual(try result.value.count, 5)
}

// Query an entity the same way Siri resolves names
func testFindCalendarByName() async throws {
    let results = try await definitions
        .entities["CalendarEntity"]
        .entities(matching: "Work")
    let name: String = try results[0].name
    XCTAssertEqual(name, "Work")
}
3 · Chain intents & verify Spotlight indexing

Pass the result of one intent directly into the next — AppIntentsTesting handles cross-intent entity transfer the same way the system does. spotlightQuery(_:) tests whether your entity donations land in the Spotlight index correctly.

// Chain: find a calendar, then create an event on it
func testCreateEventOnCalendar() async throws {
    let calendars = try await definitions
        .entities["CalendarEntity"]
        .entities(matching: "Work")

    let createResult = try await definitions
        .intents["CreateEvent"]
        .makeIntent(title: "Sprint Planning", calendar: calendars[0])
        .run()

    let event: AnyAppEntity = try createResult.value
    let addResult = try await definitions
        .intents["AddAttendee"]
        .makeIntent(event: event, email: "jada@example.com")
        .run()
    let count: Int = try addResult.value.attendeeCount
    XCTAssertEqual(count, 1)
}

// Verify Spotlight indexing
func testEventAppearsInSpotlight() async throws {
    let results = try await definitions
        .entities["EventEntity"]
        .spotlightQuery("Design Review")
    XCTAssertFalse(results.isEmpty)
    let title: String = try results[0].title
    XCTAssertEqual(title, "Design Review")
}

Most Important

AI is now a platform primitive.

Apple bakes on-device and private cloud AI directly into the OS — giving developers powerful, private, cost-free intelligence as a first-class API.

Foundation Models framework grows into agentic apps

Apps can now reason over images and text together, adapt their AI behavior on the fly as the user moves through different tasks, and optionally tap into a more powerful server model — all without managing API keys, servers, or third-party SDKs.

  • Pass CGImage, CIImage, CVPixelBuffer, or image URLs directly in a prompt via Attachment; built-in Vision tools (OCRTool, BarcodeReaderTool) work on-device with no extra setup.
  • A DynamicProfile re-evaluates instructions and tools before every model request, so switching phases (e.g. image editing → animation editing) automatically updates the active toolset without rebuilding the session.
  • The Private Cloud Compute model requires only one line to adopt: swap SystemLanguageModel() for PrivateCloudComputeLanguageModel(). It offers a 32K-token context and three reasoning levels (.light, .moderate, .deep), with no API keys or token costs.
ℹ️
The server model has a daily limit. Each user gets a fixed number of requests per day for free. If they run out, they'll need an iCloud+ subscription to get more. The on-device model has no such limit. Make sure your app handles the case where the limit is reached — and consider showing users a clear path to upgrade when it happens.
▶ Show code examples
1 · Multimodal prompt with Attachment

Pass one or more images alongside text inside the trailing closure of respond. Use orientation: when the image lacks embedded rotation metadata (e.g. frames from AVFoundation).

func compareImages(imageOne: CGImage, imageTwo: CGImage) async throws -> String {
    let session = LanguageModelSession()
    let response = try await session.respond {
        "Compare these two images using three bullet points:"
        Attachment(imageOne)
        Attachment(imageTwo, orientation: .right)
    }
    return response.content
}
2 · Dynamic Profile

Declare a DynamicProfile to switch instructions, tools, and even the underlying model based on app state. The compiler enforces that exactly one Profile is active at a time.

struct PresentationProfile: LanguageModelSession.DynamicProfile {
    var pccModel = PrivateCloudComputeLanguageModel()
    var isEditingImage = true
    var isEditingAnimation = false

    var body: some LanguageModelSession.DynamicProfile {
        if isEditingImage {
            Profile { ImageEditingInstructions() }
        } else if isEditingAnimation {
            Profile { AnimationEditingInstructions() }
                .model(pccModel)
                .temperature(0.2)
                .reasoningLevel(.light)
        } else {
            Profile { PresentationDynamicInstructions() }
                .temperature(0.8)
        }
    }
}

let session = LanguageModelSession(profile: PresentationProfile())
3 · Private Cloud Compute session

Route a session through PCC for a 32K context and deep reasoning. Check availability before use and fall back to the on-device model on unsupported versions.

// One-line swap: the rest of your session code is unchanged.
let session = LanguageModelSession(model: PrivateCloudComputeLanguageModel())

// Check availability before prompting.
let model = PrivateCloudComputeLanguageModel()
switch model.availability {
case .available:
    // Show your intelligence UI.
case .unavailable(.deviceNotEligible):
    // Show an alternative UI.
default:
    break
}

// Enable deep reasoning for complex, multi-step tasks.
let response = try await session.respond(
    to: "What are the tradeoffs in this architecture?",
    contextOptions: ContextOptions(reasoningLevel: .deep)
)
