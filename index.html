Siri becomes personal, contextual, and app-aware.

Siri now works with Apple Intelligence, App Intents, and app-provided schemas so it can understand user context and act across apps more naturally.

Personal Intelligence across apps

By adopting App Intents schemas, your app's content enters the Spotlight semantic index, enabling personal context understanding with attribution back to your app — and its actions become available through natural language without fixed trigger phrases.

  • Entity schemas contribute your app's content to the Spotlight semantic index so Siri can understand and surface it with attribution back to your app.
  • Intent schemas let people act on that content naturally — no specific phrases to define, and no code changes needed as Siri's language understanding evolves.
  • The View Annotations API adds on-screen awareness: map your views to entities so people can reference and act on whatever is visible on screen conversationally.

App Schemas make app content actionable

Apps describe entities and actions through schemas so Siri, Shortcuts, and Spotlight can reason about real app content instead of fixed trigger phrases. No specific phrases to define — the system handles language understanding and it improves over time without any code changes.

  • Important app objects need stable identities and readable names so Siri and Spotlight can find the same things users see in the interface.
  • Smaller, composable intents work better because the system can combine them with other app and system actions.
  • Set unsupported schema properties to nil — the system gracefully ignores them and still surfaces your content.
▶ Show code example
1 · Entity Schema

Conform your model to @AppEntity(schema:) so Siri and Spotlight understand what your content represents. Set unsupported properties to nil — the system gracefully ignores them.

@AppEntity(schema: .photos.asset)
struct PlantPhotoEntity: IndexedEntity {

    var displayRepresentation: DisplayRepresentation {
        DisplayRepresentation(
            title: "\(photo.name)",
            subtitle: "\(photo.creationDate.formatted(date: .abbreviated, time: .omitted))"
        )
    }

    static let defaultQuery = PlantPhotoEntityQuery()
    var id: Int { photo.id }

    // Required schema properties
    var creationDate: Date? { photo.creationDate }
    var assetType: PhotoAssetType? { .photo }
    var isFavorite: Bool { photo.isFavorite }
    var isHidden: Bool { photo.isHidden }

    // Not supported by this app — set to nil
    var location: PlaceDescriptor? { nil }
    var aperture: Double? { nil }
    var filter: PhotoFilterEffectType? { nil }

    var photo: PlantPhoto
    init(photo: PlantPhoto) { self.photo = photo }
}
2 · Intent Schemas

Conform your intent to @AppIntent(schema:) so people can trigger it through natural language — no fixed phrases needed. Use isAssistantOnly = true when transitioning away from an older intent.

@AppIntent(schema: .photos.openAsset)
struct OpenPlantPhotoIntent: OpenIntent {
    var target: PlantPhotoEntity

    func perform() async -> some IntentResult {
        // Navigate to the photo in the app

struct CreateAssetsIntent: AppIntent {
    // Hidden from Shortcuts during transition from an older intent
    static let isAssistantOnly: Bool = true

    @MainActor
    func perform() async throws -> some ReturnsValue<[PlantPhotoEntity]> {
        // Create and return new photo entities
    }
}
3 · Spotlight Indexing

Donate entities to a named CSSearchableIndex so the system can surface them. Implement IndexedEntityQuery to handle reindexing requests from the system.

static func donatePhotos(modelData: ModelData) async throws {
    let entities = await modelData.plantPhotoEntities
    try await CSSearchableIndex(name: "PlantJournal_Photos")
        .indexAppEntities(entities)
}

struct PlantPhotoQuery: IndexedEntityQuery {
    func reindexEntities(
        for identifiers: [PlantPhotoEntity.ID],
        indexDescription: CSSearchableIndexDescription
    ) async throws {
        let photos = try await photoStore.fetch(ids: identifiers)
        try await CSSearchableIndex(name: "PlantJournal_Photos")
            .indexAppEntities(photos)
    }

    func reindexAllEntities(
        indexDescription: CSSearchableIndexDescription
    ) async throws {
        let all = try await photoStore.fetchAll()
        try await CSSearchableIndex(name: "PlantJournal_Photos")
            .indexAppEntities(all)
    }
}

AppIntentsTesting validates Siri, Spotlight, and Shortcuts out-of-process

AppIntentsTesting is a new framework (iOS 27+, currently in beta) that lets you execute intents, query entities, and verify Spotlight indexing and view annotations through the exact same infrastructure that Siri and Shortcuts use — without UI automation.

  • Tests run in a UI Testing target against a live app process. IntentDefinitions(bundleIdentifier:) is the single entry point; every intent, entity, and enum is accessed by string name, so the test target never needs to import app types.
  • Covered scenarios include intent execution, entity queries, chaining results between multiple intents, Transferable conformance, Spotlight indexing (spotlightQuery(_:)), and on-screen view annotations (viewAnnotations()).
  • Test-only intents marked isDiscoverable = false and wrapped in #if DEBUG replace fragile UI navigation for test setup and teardown.
▶ Show code example
1 · Test target setup

Launch the app and create an IntentDefinitions instance with your bundle ID. This discovers every intent, entity, enum, and query your app registers — no imports of app types needed.

class AppIntentsTests: XCTestCase {
    let app = XCUIApplication()
    var definitions: IntentDefinitions!

    override func setUp() async throws {
        app.launch()
        definitions = IntentDefinitions(
            bundleIdentifier: "com.example.my-app"
        )
    }
}
2 · Run an intent & query entities

Look up an intent by name and call run(). The result supports @dynamicMemberLookup, so you read return-value properties by name. Entity queries work the same way — entities(matching:) simulates how Siri resolves spoken names.

// Run an intent
func testOpenFavorites() async throws {
    let result = try await definitions
        .intents["OpenFavorites"]
        .makeIntent()
        .run()
    XCTAssertEqual(try result.value.count, 5)
}

// Query an entity the same way Siri resolves names
func testFindCalendarByName() async throws {
    let results = try await definitions
        .entities["CalendarEntity"]
        .entities(matching: "Work")
    let name: String = try results[0].name
    XCTAssertEqual(name, "Work")
}
3 · Chain intents & verify Spotlight indexing

Pass the result of one intent directly into the next — AppIntentsTesting handles cross-intent entity transfer the same way the system does. spotlightQuery(_:) tests whether your entity donations land in the Spotlight index correctly.

// Chain: find a calendar, then create an event on it
func testCreateEventOnCalendar() async throws {
    let calendars = try await definitions
        .entities["CalendarEntity"]
        .entities(matching: "Work")

    let createResult = try await definitions
        .intents["CreateEvent"]
        .makeIntent(title: "Sprint Planning", calendar: calendars[0])
        .run()

    let event: AnyAppEntity = try createResult.value
    let addResult = try await definitions
        .intents["AddAttendee"]
        .makeIntent(event: event, email: "jada@example.com")
        .run()
    let count: Int = try addResult.value.attendeeCount
    XCTAssertEqual(count, 1)
}

// Verify Spotlight indexing
func testEventAppearsInSpotlight() async throws {
    let results = try await definitions
        .entities["EventEntity"]
        .spotlightQuery("Design Review")
    XCTAssertFalse(results.isEmpty)
    let title: String = try results[0].title
    XCTAssertEqual(title, "Design Review")
}

AI is now a platform primitive.

Apple bakes on-device and private cloud AI directly into the OS — giving developers powerful, private, cost-free intelligence as a first-class API.

Foundation Models framework grows into agentic apps

Apps can now reason over images and text together, adapt their AI behavior on the fly as the user moves through different tasks, and optionally tap into a more powerful server model — all without managing API keys, servers, or third-party SDKs.

  • Pass CGImage, CIImage, CVPixelBuffer, or image URLs directly in a prompt via Attachment; built-in Vision tools (OCRTool, BarcodeReaderTool) work on-device with no extra setup.
  • A DynamicProfile re-evaluates instructions and tools before every model request, so switching phases (e.g. image editing → animation editing) automatically updates the active toolset without rebuilding the session.
  • The Private Cloud Compute model requires only one line to adopt: swap SystemLanguageModel() for PrivateCloudComputeLanguageModel(). It offers a 32K-token context and three reasoning levels (.light, .moderate, .deep), with no API keys or token costs.
ℹ️
The server model has a daily limit. Each user gets a fixed number of requests per day for free. If they run out, they'll need an iCloud+ subscription to get more. The on-device model has no such limit. Make sure your app handles the case where the limit is reached — and consider showing users a clear path to upgrade when it happens.

1 · Multimodal prompt with Attachment

Pass one or more images alongside text inside the trailing closure of respond. Use orientation: when the image lacks embedded rotation metadata (e.g. frames from AVFoundation).

func compareImages(imageOne: CGImage, imageTwo: CGImage) async throws -> String {
    let session = LanguageModelSession()
    let response = try await session.respond {
        "Compare these two images using three bullet points:"
        Attachment(imageOne)
        Attachment(imageTwo, orientation: .right)
    }
    return response.content
}
2 · Dynamic Profile

Declare a DynamicProfile to switch instructions, tools, and even the underlying model based on app state. The compiler enforces that exactly one Profile is active at a time.

struct PresentationProfile: LanguageModelSession.DynamicProfile {
    var pccModel = PrivateCloudComputeLanguageModel()
    var isEditingImage = true
    var isEditingAnimation = false

    var body: some LanguageModelSession.DynamicProfile {
        if isEditingImage {
            Profile { ImageEditingInstructions() }
        } else if isEditingAnimation {
            Profile { AnimationEditingInstructions() }
                .model(pccModel)
                .temperature(0.2)
                .reasoningLevel(.light)
        } else {
            Profile { PresentationDynamicInstructions() }
                .temperature(0.8)
        }
    }
}

let session = LanguageModelSession(profile: PresentationProfile())

Bring your own AI model, run it entirely on device

Core AI is a new framework that lets you ship any custom AI model inside your app — no server, no API key, no usage costs. The model runs locally on the device's CPU, GPU, and Neural Engine, with a straightforward Swift API to load it and run it.

  • You convert your model (e.g. from PyTorch) into an .aimodel file using Apple's Python tools, add it to your Xcode project, and load it in a few lines of Swift. Xcode shows you the model's inputs, outputs, and size directly in the editor.
  • The first time the model loads, the device compiles it for its own hardware — this is automatic but can be slow for large models. You can move most of that work to build time using the coreai-build command-line tool, so users don't wait on first launch.
  • The system caches the compiled model after the first run, so subsequent loads are instant. You can also pre-specialize the model (e.g. right after download) so the delay never happens in the middle of a user flow.
▶ Show code examples
1 · Load and run a model

Load an .aimodel file and get a reference to its inference function. Core AI automatically specializes the model for the current device's hardware.

import CoreAI

// Specialize the model for this device and load it.
let model = try await AIModel(contentsOf: urlOfModel)

// Load the inference function by name (check the Xcode model viewer for names).
guard let function = try model.loadFunction(named: "main") else {
    // Handle case where expected function is not found.
}

// Create an NDArray matching the model's expected input shape and type.
var input = NDArray(shape: [3, 4], scalarType: .float32)

// Write your data into the array.
var mutableView = input.mutableView(as: Float.self)
if let elements = mutableView.contiguousElements {
    writeInputData(into: elements)
}

// Run inference.
var outputs = try await function.run(inputs: ["input": input])

// Read the output.
if let predictionValue = outputs.remove("prediction"),
   let prediction = predictionValue.ndArray {
    processOutput(prediction.view())
}
2 · Pre-specialize to avoid first-launch delays

Call specialize right after downloading a model so it's ready before the user needs it. On subsequent launches, AIModelCache.default returns the cached version instantly.

func loadModel(from modelURL: URL) async throws -> AIModel {
    let cache = AIModelCache.default

    // Return immediately if the model is already specialized and cached.
    if let model = try cache.model(for: modelURL, options: .default) {
        return model
    }

    // No cache yet — inform the user and specialize now.
    await MainActor.run { informUser("Preparing AI features. This may take a while…") }

    return try await AIModel(contentsOf: modelURL, options: .default)
}
3 · Ahead-of-time compilation (build time)

Run coreai-build at build time to do most of the heavy lifting on your Mac instead of on the user's device. At runtime, pick the right compiled asset for the current device architecture.

# Compile the model for iOS at build time (requires Metal Toolchain).

Test your AI features like you test your code

AI features are probabilistic — they don't always give the same answer twice. Evaluations is a new framework that brings structured, repeatable testing to model-driven features, so you can catch regressions, compare prompts, and ship with confidence. MLX separately expands what you can do locally on Apple Silicon — running and fine-tuning large models across one or multiple Macs.

  • An evaluation is a dataset of prompts + expected outputs, a subject (your feature), evaluators that score each response, and an aggregation step — all running as a Swift Testing test via @Test(.evaluates(...)), with results shown directly in Xcode's Report navigator.
  • For agentic features, TrajectoryExpectation lets you assert not just what the model said, but which tools it called, in what order, and with what arguments — catching silent failures like a wrong tool or bad argument value.
  • MLX now runs agentic workflows (e.g. OpenCode) entirely on-device, and can distribute inference or fine-tuning across multiple Macs over Thunderbolt using RDMA — letting a few machines on your desk replace expensive cloud infrastructure.
▶ Show code examples
1 · Basic evaluation with metrics

Define a dataset, a subject (your feature), evaluators that score each response, and run it as a Swift Testing test. Results appear in Xcode's Report navigator.

import Evaluations
import FoundationModels
import Testing

struct LetterCountEvaluation: Evaluation {
    let dataset = ArrayLoader(samples: [
        ModelSample(prompt: "Count the letter 'r' in 'strawberry'.", expected: 3),
        ModelSample(prompt: "How many a's are in 'banana'?", expected: 3),
    ])

    let exactMatch = Metric("ExactMatch")

    func subject(from sample: ModelSample<Int>) async throws -> ModelSubject<Int> {
        let session = LanguageModelSession()
        let response = try await session.respond(to: sample.prompt, generating: Int.self)
        return ModelSubject(value: response.content, transcript: session.transcript.structuredTranscript)
    }

    var evaluators: Evaluators {
        Evaluator { input, subject in
            guard let expected = input.expected else { return exactMatch.ignore() }
            return subject.value == expected ? exactMatch.passing() : exactMatch.failing()
        }
    }

    func aggregateMetrics(using aggregator: inout MetricsAggregator) {
        aggregator.computeMean(of: exactMatch)
    }
}

struct LetterCountTests {
    static let evaluation = LetterCountEvaluation()

    @Test(.evaluates(Self.evaluation))
    func letterCounting() async throws {
        let result = EvaluationContext.current.result
        let score = result.aggregateValue(.mean(of: Self.evaluation.exactMatch))
        #expect(score > 0.8)
    }
}
2 · Tool-calling evaluation with trajectory expectations

For agentic features, assert which tools the model must call, in what order, and with what argument values. Supports ordered, unordered, and disallowed expectations.

let toolsAllPass = Metric("ToolsAllPass")
let toolsPercentagePass = Metric("ToolsPercentagePass")

// "Start movie night" must dim lights first, then turn on the TV — order matters.
ModelSample(
    prompt: "Start movie night.",
    expectations: TrajectoryExpectation(
        ordered: [
            ToolExpectation("set_lights", arguments: [
                .exact(argumentName: "room", value: .string("living room")),
                .range(argumentName: "brightness", minimum: 5, maximum: 40),
            ]),
            ToolExpectation("set_tv", arguments: [
                .exact(argumentName: "power", value: .string("on")),
            ]),
        ]
    )
)

// Score tool calls: ToolsAllPass is a strict pass/fail,
// ToolsPercentagePass gives a 0–1 improvement signal.
var evaluators: Evaluators {
    ToolCallEvaluator(allPass: toolsAllPass, percentagePass: toolsPercentagePass)
}

func aggregateMetrics(using aggregator: inout MetricsAggregator) {
    aggregator.group("Tool Calling") { group in
        group.computeMean(of: toolsAllPass)
        group.computeMean(of: toolsPercentagePass)
    }
}
3 · Generate a synthetic dataset from a few seed samples

Writing hundreds of test cases by hand is impractical. Call makeSamples on your seed dataset to have the model expand it automatically, then validate and review before using it in evaluations.

// Start with a small hand-written seed set.
let seedDataset: [ModelSample<TaskItem>] = [ /* your seed samples */ ]

// Expand to 20 samples using a model. A validator rejects bad outputs during generation.
var expanded: [ModelSample<TaskItem>] = []
for try await sample in seedDataset.makeSamples(
    Prompt("Generate realistic to-do list items for a busy professional."),
    targetCount: 20,
    validator: { sample in
        guard let task = sample.expected else { return false }
        return !task.title.isEmpty
    }
) {
    expanded.append(sample)
}

Coding agents in Xcode

Xcode 27 brings a full coding agent into the IDE — a conversation sidebar where you describe what you want and the agent reads your project, writes code, runs builds, fixes errors, and shows you a diff to review, all without leaving Xcode.

  • The agent works in four modes: Explore (understand unfamiliar code), Build (implement features or prototypes), Refine (polish existing code), and Orchestrate (coordinate multi-step changes across files). You can run multiple agents in parallel in separate panes.
  • Give the agent explicit context using @filename or @Symbol in your prompt. The more specific you are — naming the files involved, the expected behavior, and which tests should pass — the better the result.
  • Every change is reviewable: the artifacts pane shows a multicolor diff for each modified file. You can undo a single response or slide back through the full conversation history to restore any prior state (requires a Git repo).

Localization agents

Agents can also translate your String Catalogs into new languages, handle plural variants, and apply Apple style guidance — all from within Xcode.

  • Works best when strings are already in String Catalogs rather than scattered through views. Ambiguous words like "Archive", "Draft", or "Charge" benefit from added context comments so the agent picks the right translation.
  • Plural rules, purchase copy, privacy text, and legal language still need a careful human review before shipping.

Swift 6.4 brings ergonomics improvements, new library types, and better testing.

Swift 6.4 brings language ergonomics improvements, improved C and Java interoperability, WebAssembly support, ownership-safe standard library types, and Swift Testing / XCTest interop. UIKit and AppKit also get practical modernization paths with Xcode agent help.

Everyday language improvements

Swift 6.4 trims common boilerplate and gives you better tools for managing API evolution.

  • anyAppleOS availability: write #available(anyAppleOS 27, *) instead of listing iOS, macOS, watchOS, and tvOS separately.
  • @diagnose attribute: attach a custom warning or error to any API — great for nudging teammates away from deprecated code without removing it yet.
  • Module selectors (::): when two frameworks share a type name, write Payments::Order instead of the full module path. Less noise, same clarity.

New Document API gives apps direct, async control over disk access

Two new protocols let you read and write files asynchronously, with built-in progress reporting.

  • Conform to WritableDocument / ReadableDocument to handle file I/O asynchronously — no blocking the main thread.
  • Progress reporting works automatically via Foundation's Subprogress — users see a real save indicator without any extra code.
  • Use DocumentCreationSource to offer multiple templates in the same app — e.g. Blank vs. From Template — each with its own NewDocumentButton.
▶ Show code example
WritableDocument + DocumentCreationSource

Conform to WritableDocument/ReadableDocument for async disk access, and declare multiple creation sources for template support.

struct NoteDocument: WritableDocument, ReadableDocument {
    var text: String

    init(configuration: ReadConfiguration) async throws {
        let data = try configuration.file.regularFileContents
            .unwrap(throwing: CocoaError(.fileReadCorruptFile))
        self.text = String(decoding: data, as: UTF8.self)
    }

    func fileWrapper(configuration: WriteConfiguration) async throws -> FileWrapper {
        FileWrapper(regularFileWithContents: Data(text.utf8))
    }
}

struct ContentView: View {
    var body: some View {
        DocumentGroup(newDocument: NoteDocument(text: "")) { file in
            TextEditor(text: file.$document.text)
        } creationSources: {
            DocumentCreationSource("Blank Note") {
                NoteDocument(text: "")
            }
            DocumentCreationSource("From Template") {
                NoteDocument(text: "# Title\n\nStart writing…")
            }
        }
    }
}

Toolbar APIs give precise control over visibility, overflow, and auto-minimizing

Four new modifiers let you decide exactly what happens to toolbar items when space gets tight.

  • visibilityPriority — mark a group as low priority so it collapses first when the toolbar gets crowded.
  • toolbarOverflowMenu — always move secondary actions into an overflow menu, regardless of available space.
  • topBarPinnedTrailing — keep a critical action (like Share) pinned to the right edge at all times.
  • toolbarMinimizeBehavior(.onScroll) — auto-hide the navigation bar while scrolling to give content more room.

Toolbar visibility, overflow, and auto-collapse

Combine the new modifiers to control which items survive layout compression and how the navigation bar behaves while scrolling.

NavigationStack {
    List(items) { item in ItemRow(item: item) }
    .toolbar {
        ToolbarItemGroup(placement: .topBarLeading) {
            FilterButton()
            SortButton()
        }
        .visibilityPriority(.low) // collapse first when space is tight

        ToolbarItem(placement: .topBarPinnedTrailing) {
            ShareLink(item: shareURL) // always visible, pinned to trailing
        }
    }
    .toolbarOverflowMenu {
        // always in overflow regardless of available space
        ExportButton()
        PrintButton()
    }
    .toolbarMinimizeBehavior(.onScroll)
    .navigationTitle("Items")
}

Reordering, swipe actions, and presentations work across more containers

Drag-to-reorder, swipe actions, and confirmation dialogs all work in more places — not just List.

  • Drag-to-reorder now works in grids, sections, and custom layouts using the same API as List. Also new on watchOS.
  • Add .swipeActionsContainer() to any ScrollView to enable swipe actions on every row inside it.
  • Confirmation dialogs now use the same item-binding pattern as sheets — set a value, the dialog appears automatically.
  • You can also drag multiple items at once and customize the drag lifecycle to match your app's rules.

Reordering in a LazyVGrid

The reorderable API that works on List rows now applies to grid cells and other containers.

@State private var items = Item.sampleItems

LazyVGrid(columns: [GridItem(.adaptive(minimum: 120))]) {
    ForEach($items) { $item in
        ItemCard(item: item)
    }
    .reorderable(with: $items)
}
Swipe actions on ScrollView + item-bound alert

swipeActionsContainer enables swipe actions on any view inside a ScrollView. Confirmation alerts now use the same item-binding pattern as sheets.

@State private var itemToDelete: Item?

ScrollView {
    LazyVStack {
        ForEach(items) { item in
            ItemRow(item: item)
                .swipeActions(edge: .trailing) {
                    Button(role: .destructive) {
                        itemToDelete = item // triggers confirmation dialog
                    } label: {
                        Label("Delete", systemImage: "trash")
                    }
                }
        }
    }
}
.swipeActionsContainer()
.confirmationDialog(
    "Delete item?",
    item: $itemToDelete          // item-binding pattern — same as sheet(item:)
) { item in
    Button("Delete \(item.name)", role: .destructive) { delete(item) }
}

AsyncImage caches by default, @State becomes a macro, and builds get faster

AsyncImage now caches automatically, @State initializes classes only once, and Xcode 27 builds are faster — all with no code changes required.

  • AsyncImage respects HTTP cache headers by default — images are cached automatically. Use asyncImageURLSession to override the cache policy if needed.
  • @State is now a macro. Classes inside it are created once per view lifetime, not on every parent re-render.
  • ViewBuilder is now ContentBuilder internally, which noticeably speeds up Xcode 27 build times. Existing code keeps working — see TN3211 if you hit any source incompatibilities.
▶ Show code example
AsyncImage with custom URLSession cache

By default, AsyncImage now uses HTTP caching. Use asyncImageURLSession to supply a custom cache policy.

// Default — HTTP cache headers respected automatically, no code needed
AsyncImage(url: thumbnailURL)

// Custom 100 MB memory + 500 MB disk cache
let config = URLSessionConfiguration.default
config.urlCache = URLCache(
    memoryCapacity: 100_000_000,
    diskCapacity:   500_000_000
)
AsyncImage(url: thumbnailURL)
    .asyncImageURLSession(URLSession(configuration: config))
@State macro — lazy class initialization

@State is now a Swift macro. Class instances are created once per view lifetime — not on every parent re-render.

struct PhotoBrowser: View {
    // PhotoViewModel() is now created lazily — once per view lifetime.
    // Before the macro, it was re-created on every parent re-render.
    @State private var model = PhotoViewModel()

    var body: some View {
        PhotoGrid(model: model)
    }
}

Lazy stacks and scrolling internals are documented and optimizable

Apple has documented how lazy stacks work under the hood — useful reading if you're debugging layout jumps or scroll glitches.

  • Size estimation: lazy stacks estimate row heights before rendering. Provide accurate estimates to avoid layout jumps when cells load in.
  • Prefetching: use onPrefetchTask to kick off network requests before a cell scrolls into view.
  • Programmatic scrolling: use scrollPosition(id:) rather than manual offset math — it handles dynamic sizing and avoids position drift after reloads.
  • State in recycled cells: don't put view-lifetime state in @State inside cells that get recycled — use model-layer state with a stable ID instead.

Advanced graphics effects compose shaders, timelines, and alignment guides

Rich custom visuals are achievable by composing simple SwiftUI primitives — no custom UIView or Metal code needed.

  • Layer shaders: use Shader to apply Metal-powered effects (gradients, distortion, color shifts) directly on any SwiftUI view.
  • Time-driven animation: pair TimelineView with Canvas to keep effects updating at display refresh rate, driven by elapsed time.
  • Alignment guides: custom alignment guides let you anchor floating elements — like timestamps or tooltips — precisely relative to any other view.
  • The key idea: chain small, focused modifiers instead of one large drawing block — easier to reuse, test, and swap out.
